## 0. Project at a glance

**Foundation vs. Slop** — an isometric low-poly 3D game. Premise: the SCP Foundation
fights "slop" entities (deliberately ugly, uncanny-valley monsters) generated by
SCP-9191, a rogue monster-generating AI.

Current state: an **explorable, procedurally-generated dungeon** — WFC rooms & corridors
built from the Kenney Prototype Kit, fog of war, a model hero (Kenney `figurine`) that
walks around with wall collision, and a follow/free-pan isometric camera. The slop
enemies are **shelved** (code kept, not compiled) pending a later pass.

### Run it
```bash
cargo run              # dev; asset root auto-resolves via CARGO_MANIFEST_DIR
cargo run --release    # smoother
```
Running the raw binary needs the asset root: `BEVY_ASSET_ROOT=. ./target/debug/foundation_vs_slop`

### Controls
- **WASD** — move the hero (camera-aligned)
- **Tab** — toggle camera Follow ↔ FreePan
- **Arrow keys / middle-mouse drag** — pan (FreePan mode)
- **Mouse wheel** — zoom

- **Rust 1.96.1**, edition **2024**. Bevy 0.19 requires rustc ≥ 1.95 — we had to

- Platform: macOS (Apple M5), **Metal** backend. 32 GB.
- `Cargo.toml` dev profile: `opt-level = 1` for our crate, `3` for deps (Bevy's

- **Only dependency:** `bevy = "0.19.0"`. Everything else (PRNG, WFC) is hand-rolled to
  stay dependency-light.

### Bevy 0.19 API gotchas (these bit us repeatedly — the engine drifted a lot)
| Old / expected | 0.19 reality |
|---|---|
| `SceneRoot(handle)` | **`WorldAssetRoot(handle)`** — glTF scenes load as `Handle<WorldAsset>` (from `bevy_world_serialization`). It's in the prelude and `#[require(Transform, Visibility)]`. |
| `AmbientLight` as a `Resource` | `AmbientLight` is now a per-camera **component**; the scene-wide one is **`GlobalAmbientLight`** (resource). |
| `DirectionalLight { shadows_enabled }` | **`shadow_maps_enabled`** (same rename on `PointLight`). |
| `EventReader<MouseWheel>` | Buffered events are **"Messages"** now → `MessageReader<T>`, or better use the **`AccumulatedMouseScroll` / `AccumulatedMouseMotion`** resources (import from `bevy::input::mouse`, NOT in prelude). |
| `ScalingMode` in `bevy::render::camera` | now **`bevy::camera::ScalingMode`**. |
| `PbrBundle` | wrapper components **`Mesh3d(..)`**, **`MeshMaterial3d(..)`**; camera = `Camera3d::default()` + `Projection::from(OrthographicProjection { .., ..OrthographicProjection::default_3d() })`. |
| `Query::single_mut()` infallible | returns `Result` now; we use the **`Single<T>`** system param (panics/skips if not exactly one — fine for guaranteed-singleton entities). |

**How to check an API in 0.19:** read the crate source directly at
`~/.cargo/registry/src/index.crates.io-*/bevy_*-0.19.0/`. Web docs lag; the source is truth.

## 2. Architecture — one Bevy plugin per domain

`src/main.rs` registers plugins in this order (**order matters**, see below):
```
DungeonPlugin → WorldPlugin → CameraPlugin → PlayerPlugin → FogPlugin
```

| File | Plugin | Responsibility |
|---|---|---|
| `wfc.rs` | — (engine-free lib) | Wave Function Collapse over a coarse room graph. No Bevy deps. |
| `dungeon.rs` | `DungeonPlugin` | Runs WFC, connectivity, expands to a fine tile grid, spawns Kenney models, owns the `Dungeon` resource (walkability + collision). |
| `world.rs` | `WorldPlugin` | Lighting only (`GlobalAmbientLight` kept low + a weak `DirectionalLight`). No ground plane — the dungeon lays its own floor. |
| `camera.rs` | `CameraPlugin` | Isometric orthographic focus-point camera; Follow/FreePan; zoom. Exports `SCREEN_FORWARD`/`SCREEN_RIGHT` (the camera-aligned movement basis). |
| `player.rs` | `PlayerPlugin` | Spawns the `figurine` hero + torch light; camera-aligned WASD → `Dungeon::resolve_move`. |
| `fog.rs` | `FogPlugin` | Per-cell reveal memory; toggles tile `Visibility`. |
| `enemies.rs`, `combat.rs` | *(shelved)* | Slop spawner + contact despawn. **Not** declared in `main.rs` (files on disk, not compiled). |

### Plugin ordering / resource-timing trick
`DungeonPlugin::build()` runs the (pure, no-asset) WFC generation and **inserts the
`Dungeon` resource during `build()`** — not in a Startup system. That way the resource
exists before any Startup system runs, so `PlayerPlugin` (spawn at `dungeon.spawn`) and
`FogPlugin` (sizes `FogGrid` from dungeon dims, read in *its* `build()`) can rely on it
without cross-plugin Startup ordering games. **Consequence:** DungeonPlugin must be
registered before FogPlugin.

## 3. Coordinate & grid conventions (memorize these)

- **Directions** (indices into every `[_; 4]`): `N=0 (−Z)`, `E=1 (+X)`, `S=2 (+Z)`,
  `W=3 (−X)`. `opposite`: N↔S, E↔W.
- **Grid** is row-major: `cells[y * width + x]`. An `IVec2(x, y)` has `y` = grid row =
  world **Z**.
- **World ↔ cell:** `world = cell * TILE_SIZE`; `world_to_cell = round(pos / TILE_SIZE)`.
  Cell `c` occupies world `[c−0.5, c+0.5] * TILE_SIZE`, centered on `c * TILE_SIZE`.
- `TILE_SIZE = 1.0` (Kenney floor tile is exactly 1×1).
- Socket 90° rotation helper `rotate_cw([N,E,S,W]) = [W,N,E,S]` (i.e. `[a[3],a[0],a[1],a[2]]`).

## 4. The WFC system (`wfc.rs` + `dungeon.rs`)

We generate in **two levels** — this is the key design decision:

### 4a. Coarse room-graph WFC (`wfc.rs`)
- Grid of **room slots**. Each slot collapses to a *connection prototype*: a room with a
  **Link** or **Wall** on each edge (the `open: [bool; 4]` field, `true` = corridor
  links to that neighbour), or `EMPTY` rock (all Wall).
- **Adjacency rule = symmetric equality** on the shared edge (Link meets Link, Wall meets
  Wall). This single rule guarantees corridors always join two rooms and never dead-end
  into rock.
- Prototypes (kind `Floor` = room, `Solid` = empty rock), each expanded over its distinct
  rotations: `EMPTY`, dead-end (1 link), corridor-through (2 opposite), corner (2
  adjacent), tee (3), cross (4). **Weights** bias the mix (in `build_prototypes`):
  `EMPTY 6, deadend 1.2, corridor 2.5, corner 2.5, tee 1.2, cross 0.6`. Tune these to
  change density/negative-space.
- Algorithm: min-entropy observe → weighted collapse → constraint propagation with **u32
  bitmasks** (`support[dir][a]` = mask of prototypes allowed on the `dir` side of `a`).
  On contradiction, **restart with an offset seed** up to `MAX_ATTEMPTS` (20), then
  **panic loudly** (no silent fallback — matches the repo's one-path rule).
- Deterministic via a hand-rolled **xorshift64 `Rng`** (`pub(crate)`, shared with
  `dungeon` for room sizing — one PRNG impl). Seeded from `DUNGEON_SEED` (fixed → same
  dungeon every run; change the const to reroll).

### 4b. Expansion to the fine grid (`dungeon.rs::Dungeon::generate`)
- Coarse grid `COARSE_W × COARSE_H = 7 × 7`; each slot → a `BLOCK × BLOCK = 7 × 7` patch →
  fine grid **49 × 49** (much larger than the ~12-tile viewport → scrollable).
- **Connectivity pass** (`largest_room_component`): flood-fill room slots across Link
  edges, keep only the largest component; the rest become rock. Guarantees every room is
  reachable (needed for "explore the dungeon").
- Each kept slot → a **centered rectangular room** of seeded size `ROOM_MIN..ROOM_MAX`
  (4–5) inside its block, leaving a rock margin (so rooms don't merge → negative space).
- Each Link edge → a **2-wide corridor** carved between the two block centers. (2-wide,
  not 1 — see the clipping bug; the hero is 0.7 wide and wouldn't fit a 1-wide corridor.)
- `Dungeon` stores a flat `walkable: Vec<bool>` + `spawn` (block center of the kept room
  nearest the coarse center).

### 4c. Tile instantiation (`spawn_tiles`)
- One `floor-square.glb` per floor cell.
- Walls only on **floor↔rock (or off-grid) edges** — the room perimeter. No inter-cell
  walls (that was the old "maze" look).
- Per cell, a **bitmask of walled edges** → greedily consume adjacent pairs as
  `wall-corner.glb` L-pieces (rotated NE→NW→SW→SE = quarter-turns 0→3), remaining single
  edges as straight `wall.glb`.
- Every piece is tagged `Tile { cell }` so fog reveals a cell's geometry as a unit.
- All tiles spawn `Visibility::Hidden` (fog reveals them).

## 5. Kenney Prototype Kit — asset facts (CC0)

Path: `assets/kenney_prototype-kit/Models/GLB format/*.glb` (145 models).

**Measured geometry (with node transforms applied — see Bug #4):**
- `floor-square.glb`: X/Z `[−0.5, 0.5]`, flat at y=0. → `TILE_SIZE = 1.0`.
- `wall.glb`: **0.2 thick (X `[−0.1, 0.1]`, centered on origin)**, 1.0 long (Z), 1.0 tall
  (Y `[0, 1]`).
- `wall-corner.glb`: L-piece; arms **flush inside the tile edges** — east arm X `[0.3,
  0.5]` (outer face at +0.5), north arm Z `[−0.5, −0.3]`. Covers **N+E**; rotate for the
  other 3 corners.
- `figurine.glb`: a **node hierarchy** (legs/torso/arms/head, each with node
  translations). True scene-space bounds: **feet at y=0**, 0.7 tall, footprint half
  **X 0.25 (arms) × Z 0.10 (body)**. At scale 1.4 → 0.35 × 0.14, ~0.98 tall.

**Convention that matters:** Kenney walls/corners are built **flush inside the tile edge**
(outer face on the tile boundary, body inside). Straight `wall.glb` is *centered on its
origin*, so to sit flush you must **inset it by its half-thickness (0.1)** — see Bug #6.

## 6. Collision system (`dungeon.rs`) — subtle, read before touching

Walls sit flush → a walled cell's walkable area is inset by `WALL_THICKNESS = 0.2`; the
wall's **inner face is 0.3 from the cell center**.

- `is_solid(x, z)` — ground truth: a world point is solid if its cell is rock/off-grid,
  **or** it's within a walled-edge strip (local coord past the inner face). This encodes
  the exact flush-wall geometry.
- `resolve_move(pos, delta, half: Vec2)` — axis-separated (so the hero slides along
  walls). Per axis: move, then if the **leading edge** (`± half-extent`, sampled at 3
  points across the box) is solid, snap it to the wall face:
  `face = (round(edge) + 0.5·dir)·T − dir·WALL_THICKNESS`.
- `half` = the player's **box half-extents** (X, Z), matching the figurine footprint ×
  scale = `(0.35, 0.14)` — arms wide, body thin. The model touches but never penetrates.
- **Assumption:** per-frame step < wall thickness (speed 5 × dt ≈ 0.08 ≪ 0.2), so the
  edge always stops inside the wall slab (no tunneling). If you raise speed or dt spikes,
  add sub-stepping or cap delta.

## 7. Fog, camera, player (quick reference)

- **Fog** (`fog.rs`): `FogGrid` of `Hidden|Revealed` per fine cell. Each frame, reveal the
  disc within `REVEAL_RADIUS = 5` of the player's cell (permanent). When the revealed set
  grows, set matching `Tile` entities' `Visibility` to `Visible`. Unexplored = pure black.
  The **currently-visible vs remembered gradient comes for free** from the player's torch
  `PointLight` + low `GlobalAmbientLight` (no per-tile material work).
- **Camera** (`camera.rs`): one `CameraRig { focus, mode, height }`. Camera transform each
  frame = `focus + ISO_OFFSET` looking at `focus`. Follow lerps `focus` → player
  (frame-rate-independent exp smoothing). FreePan moves `focus` via arrows / MMB drag.
  `Tab` toggles; wheel adjusts `height` (ortho `FixedVertical` scaling). Movement basis
  `SCREEN_FORWARD=(−1,0,−1)`, `SCREEN_RIGHT=(1,0,−1)` lives here.
- **Player** (`player.rs`): `figurine` `WorldAssetRoot` at `dungeon.spawn_world()` (y=0,
  scale 1.4), `Player` + `MoveSpeed`, child `PointLight` torch. Movement reads WASD along
  the screen basis, normalizes, `delta = dir·speed·dt`, then
  `dungeon.resolve_move(pos, delta, PLAYER_HALF_EXTENTS)`.

## 8. Bugs & Challenges (chronological)

### Bug #1 — Bevy 0.19 wouldn't compile (toolchain)
**Symptom:** `bevy@0.19.0 requires rustc 1.95.0`; we had 1.94.
**Solution:** `rustup update stable` → 1.96.1.
**Lesson:** check the crate's `rust-version` before assuming a build failure is code.

### Bug #2 — a cascade of 0.19 API renames
**Symptom:** `AmbientLight is not a Resource`, `no field shadows_enabled`, `cannot find
SceneRoot`, `EventReader not found`, `AccumulatedMouse* not in scope`.
**Root cause:** Bevy 0.19 reworked scenes (BSN), lights, and buffered events.
**Solution:** see the table in §1. **Read the registry source** to find the new names.
**Lesson:** for a brand-new Bevy, verify every non-trivial component/param against source.

### Bug #3 — assets not found when running the binary
**Symptom:** `Path not found: target/debug/assets/...`.
**Root cause:** the raw binary resolves assets relative to the executable; `cargo run`
uses `CARGO_MANIFEST_DIR`.
**Solution:** run via `cargo run`, or set `BEVY_ASSET_ROOT`.

### Bug #4 — the hero floated above the floor
**Symptom:** figurine hovered ~0.5 above the ground.
**Initial hypothesis:** feet at y=−0.3 (from the mesh POSITION accessor), so lift by +0.3.
**Investigation:** dumped the glTF node tree — `figurine` is a hierarchy whose meshes are
lifted by **node translations** (legs at +0.25, torso at +0.25…).
**Root cause:** mesh-local bounds ≠ scene-space bounds. The true feet are at **y=0**; my
+0.3 offset (×1.4 scale) was pushing it into the air.
**Solution:** `FIGURINE_FOOT_OFFSET = 0.0`.
**Lesson:** **always apply node transforms** when measuring a glTF's real bounds. Mesh
POSITION min/max is in *local* space and misleading for rigged/parented models.

### Bug #5 — dungeon was a wall maze, no negative space
**Symptom:** dense thin walls everywhere; 1-cell "rooms".
**Root cause:** the first WFC was a *per-tile passage* model that put walls between
adjacent floor cells and mediated every floor↔rock boundary with wall tiles.
**Solution:** rewrote to the **coarse room-slot WFC + expansion** (§4). Rooms & corridors
with rock between.
**Lesson:** choose the WFC granularity to match the output you want. Per-tile passage →
organic corridors/noise; coarse room-slot → clean rooms & corridors.

### Bug #6 — wall corners didn't line up
**Symptom:** a 0.1 jog/notch where straight walls met corner pieces.
**Investigation:** measured exact geometry. Corner arms are **flush inside** the tile edge
(outer face at ±0.5); straight `wall.glb` is **centered on its origin**, and I was placing
it *on the grid line* (i+0.5) → it straddled the boundary [i+0.4, i+0.6], 0.1 proud of the
corners.
**Solution:** inset straight walls by their half-thickness so the **outer face lands on
±0.5**, matching the corner arms (`WALL_HALF_THICKNESS = 0.1`).
**Lesson:** modular kits assume a specific placement convention (here: flush inside).
Measure the reference piece (the corner) and match everything to it.

### Bug #7 — the hero clipped through walls
**Symptom:** the figurine pushed its body into/through walls.
**Root cause (three compounding):** (a) collision clamped at the **grid line** (i+0.5),
but flush walls have their inner face at i+0.3 → the body entered the 0.2 wall; (b) a
single 0.3 radius was **smaller than the 0.35 arm reach**; (c) the 0.7-wide figurine
**didn't fit** a 1-wide corridor (0.6 clear).
**Solution:** wall-aware collision (`is_solid` + snap to the flush inner face), **per-axis
box half-extents** matching the footprint (0.35 × 0.14), and **2-wide corridors**.
**Lesson:** collision geometry must match the *rendered* wall geometry exactly; and size
passages to the actor (measure the actor's footprint × scale).

## 9. Tuning knobs (where to change behavior)

- **Dungeon shape:** `wfc.rs::build_prototypes` weights (density/negative space);
  `dungeon.rs` `COARSE_W/H`, `BLOCK`, `ROOM_MIN/MAX`, corridor width (the 2-wide carve).
- **Reroll dungeon:** `DUNGEON_SEED` (currently fixed for reproducibility).
- **Wall geometry:** `WALL_HALF_THICKNESS` / `WALL_THICKNESS` (keep = kit's 0.2 thickness).
- **Hero:** `player.rs` `PLAYER_SPEED`, `FIGURINE_SCALE`, `PLAYER_HALF_EXTENTS`
  (= footprint × scale — keep in sync if you rescale).
- **Fog:** `fog.rs` `REVEAL_RADIUS`.
- **Camera:** `camera.rs` `ISO_OFFSET`, `VIEWPORT_HEIGHT`, zoom/pan/follow rates.

## 11. Global engineering rule (from the user's CLAUDE.md — honor it)

**One execution path per feature. Fail loudly. No fallbacks, legacy shims, stubs, or
"backup" modes.** When the primary path can't produce a usable result, panic/error — don't
write a degraded substitute. (This is why WFC panics after `MAX_ATTEMPTS` instead of
emitting a broken grid, and why collision has a single authority in `Dungeon`.)

## 12. Known issues / open questions

- **Movement is camera-aligned, corridors run on world axes** → walking a "north" corridor
  needs a two-key diagonal (W+D). Options: rotate the movement basis to world axes, or add
  grid-snapped movement. (Biggest UX wart.)
- **Rare 3–4-walled cells** (dead-end pockets): a corner-piece arm can overlap a straight
  wall → possible z-fighting on the wall tops. Uncommon given room sizes; acceptable.
- **Collision assumes small per-frame steps** (no tunneling). Revisit if speed/dt changes.
- **Figurine doesn't rotate** to face movement direction.
- **Deterministic seed** — same dungeon every run (good for debugging, add reroll later).
- Enemies/combat are **shelved** (files present, not compiled/registered).
